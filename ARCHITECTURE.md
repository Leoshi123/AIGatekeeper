## 🧩 Componentes Principales

### 1. MetadataSanitizer (src/sanitizer/metadata_cleaner.py)

**Propósito**: Eliminar metadatos sensibles del código generado por IA.

**Características**:
- Limpia rutas absolutas (Windows, Linux, Mac)
- Elimina comentarios de IA (`// Thinking:`, `// Generated by...`)
- Reemplaza tokens de sesión y IDs con `[REDACTED]`
- Elimina firmas de modelos (`gpt-4`, `claude-code`, etc.)

**Tecnologías**: Regex patterns compilados

**Ejemplo de uso**:
```python
from src.sanitizer import MetadataSanitizer

sanitizer = MetadataSanitizer()
result = sanitizer.sanitize(code_original)

print(result.cleaned_code)          # Código limpio
print(result.removed_items)       # Lista de elementos removidos
```

---

### 2. LegacyShield (src/detector/zombie_detector.py)

**Propósito**: Detectar funciones obsoletas, vulnerables o peligrosas en el código.

**Características**:
- 30+ patrones de código zombi predefinidos
- Soporte para Python, JavaScript, TypeScript y patrones generales
- Clasificación por severidad: CRITICAL, HIGH, MEDIUM, LOW, INFO
- Referencias a CVEs cuando están disponibles
- Generación automática de sugerencias de refactorización

**Patrones Incluidos**:

| Lenguaje | Patrón | Severidad |
|----------|--------|-----------|
| Python | `eval()` | CRITICAL |
| Python | `exec()` | CRITICAL |
| Python | `subprocess` + `shell=True` | CRITICAL |
| Python | `pickle.loads()` | HIGH |
| Python | `yaml.load()` sin Loader | HIGH |
| JavaScript | `eval()` | CRITICAL |
| JavaScript | `document.write()` | CRITICAL |
| JavaScript | `innerHTML` | HIGH |
| General | API keys hardcodeadas | CRITICAL |
| General | Passwords hardcodeadas | CRITICAL |

**Tecnologías**: Regex + análisis línea por línea

**Ejemplo de uso**:
```python
from src.detector import LegacyShield, Severity

shield = LegacyShield()
results = shield.scan_code(code)

# Filtrar solo críticos
critical = [r for r in results if r.pattern.severity == Severity.CRITICAL]

# Bloquear si hay problemas críticos
if shield.block_critical(results):
    print("🛑 Código bloqueado por seguridad")
```

### 3. ASTExtractor (src/ast_parser/extractor.py)

**Propósito**: Reducir el contexto enviado a la IA extrayendo solo funciones relevantes.

**Características**:
- Detección automática de lenguaje por extensión
- Extracción de funciones relevantes basada en descripción de tarea
- Inferencia de dependencias entre funciones
- twofallback con regex cuando tree-sitter no está disponible

**Reducción típica**: 50-80% del código original

**Tecnologías**: Regex (fallback) / tree-sitter (avanzado, no implementado)

**Ejemplo de uso**:
```python
from src.ast_parser import ASTExtractor

extractor = ASTExtractor(use_tree_sitter=False)
pruned = extractor.prune("archivo.py", "optimizar función login")

print(f"Reducción: {extractor.get_stats(pruned)['reduction_percent']}%")
# Output: Reducción: 65%
```

---

### 4. AIAgentWrapper (src/wrapper/agent_wrapper.py)

**Propósito**: Envolver llamadas a agentes de IA (Claude Code, OpenCode) con sanitización automática.

**Características**:
- Sanitización de input antes de enviar a la IA
- Sanitización de output antes de escribir en disco
- Poda opcional de contexto usando ASTExtractor
- Bloqueo automático si se detectan vulnerabilidades críticas
- Verificación de disponibilidad de agentes

**Tecnologías**: subprocess para ejecutar agentes

**Ejemplo de uso**:
```python
from src.wrapper import AIAgentWrapper, WrapperConfig

config = WrapperConfig(
    agent_command="claude",
    sanitize_input=True,
    sanitize_output=True,
    prune_context=True,
    block_critical=True,
)

wrapper = AIAgentWrapper()
result = wrapper.run("Optimizá la función login", config)

print(result.stdout)           # Output sanitizado
print(result.security_issues_found)  # Cantidad de issues encontrados
```

## 🎯 Decisiones de Diseño

### ¿Por qué regex en lugar de AST real?

**Decisión**: Usar regex como implementación primary con fallback a tree-sitter.

**Razón**:
1. **Simplicidad**: No requiere dependencias externas pesadas
2. **Portabilidad**: Funciona en cualquier entorno Python
3. **Velocidad**: Suficiente para archivos de código típicos (<10k líneas)
4. **Mantenibilidad**: Patrones regex son fáciles de entender y modificar

** trade-off**: Menos preciso que AST en casos edge, pero más fácil de extender.

### ¿Por qué no monkey-patching?

**Decisión**: Wrapper basado en subprocess, no interceptación de funciones.

**Razón**:
1. **Seguridad**: No modifica el comportamiento del agente de IA
2. **Portabilidad**: Funciona con cualquier agente que exponga CLI
3. **Debugging**: Fácil de auditar y trazar

### Sistema de severidad

| Nivel | Significado | Acción por defecto |
|-------|-------------|-------------------|
| CRITICAL | Vulnerabilidad explotable | Bloquear output |
| HIGH | Código peligroso | Advertir |
| MEDIUM | Deprecated o ineficiente | Advertir |
| LOW | sugerencia de mejora | Informar |
| INFO | Información | Ignorar |
